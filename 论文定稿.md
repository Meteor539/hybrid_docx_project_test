 摘  要
论文格式检查是学术写作中的重要环节。针对人工校对效率低、易遗漏等问题，本文设计并实现了一个基于Python的Word文档格式检查系统。
系统以页面版面识别为主线，在Word转PDF后提取页面文本、坐标及局部空间关系，对页码、页眉、章节分页、图题表题位置和公式编号位置等版面问题进行自动判断。在此基础上，系统结合Word文档对象解析，对摘要、关键词、目录、标题、引文、参考文献以及字体、字号、对齐方式和行距等内容进行辅助检查，并实现了规则调度、结果合并、问题定位和图形界面展示功能。
测试结果表明，系统能够较好地完成多数常见论文格式问题的自动检查，在文档结构检查、样式属性检查、页码页眉检查和结果可视化展示方面具有较好的实用效果。本文完成了一个以页面识别为核心、结合文档对象分析的论文格式检查原型系统。
关键词：格式检查；Word文档；页面分析；可视化定位


 Abstract
Format checking is an important part of academic writing. To improve the low efficiency and omission risk of manual proofreading, this paper designs and implements a Python-based Word document format checking system.
The system takes page-layout recognition as the main line. After converting a Word document into PDF, it extracts page text, coordinates, and local spatial relations to detect page numbers, headers, chapter pagination, caption positions, and formula numbering positions. It also combines Word document object parsing to check abstracts, keywords, tables of contents, headings, citations, references, and style attributes such as fonts, font sizes, alignment, and line spacing. In addition, unified rule scheduling, result merging, issue localization, and graphical presentation are implemented.
The test results show that the system can effectively detect most common paper-format problems and performs well in structure checking, style checking, page-number and header checking, and visual result presentation. This paper completes a prototype paper-format checking system centered on page recognition and supported by document object analysis.
Key Words：Format checking；Word Document；Page Layout Analysis；Visual Localization

 第1章 绪论
1.1 课题研究背景
随着高校论文写作逐渐全面电子化，论文排版与格式规范问题也变得越来越突出。对于学位论文和课程论文等常见学术文档而言，除研究内容本身外，论文格式是否符合学校统一要求，同样是质量评价中的重要组成部分。这类论文通常包含封面、摘要、目录、正文、参考文献、致谢等多个部分，不同部分在字体、字号、对齐方式、行距、页眉、页码以及图表公式编号等方面均有较为明确的规范要求。如果这些内容依靠人工逐项核对，不仅工作量较大，而且容易因为疏忽而遗漏问题。
从实际写作过程来看，学生在论文撰写中经常会遇到格式设置不统一、目录与正文不一致、页码位置不正确、标题层级错误、图题表题位置不规范、参考文献编号混乱等问题。尤其需要注意的是，在论文经过多次修改以及反复排版处理之后，原本已经调整得比较完善的格式可能会再一次发生变动，这样一来，后期人工校对工作的负担便得到了进一步的加重。所以，去研究一种可以自动发现论文格式问题的辅助检查系统，有着较强的现实意义。
另一方面，近年来文档处理、智能排版分析以及文本信息提取等技术持续发展，这就给论文格式的自动检查工作打下了技术基础。借助Word文档对象解析技术，能够较为便利地去获取段落、标题、字体以及样式这些结构化信息；借助PDF页面分析技术，则能够进一步获取页面里面的文本位置和版面布局方面的特征信息。把这两类信息进行结合，就有希望实现对论文的结构类规则以及版面类规则的联合检查工作，这样一来，就能提升格式检查的覆盖范围和准确性。
基于上述背景，本文选择论文格式自动检查作为研究对象，设计并实现一个基于Python的Word文档格式检查系统，并以学校论文规范作为主要应用场景展开验证，以期为论文排版规范化提供一种可行的辅助工具。
1.2 课题研究目的与意义
本课题的研究目的，是围绕论文格式检查这一实际需求，设计并实现一个能够对Word文档进行自动分析、自动检查和结果可视化展示的软件系统。系统不仅需要对摘要、关键词、目录、标题、参考文献等内容结构和样式属性进行检查，还需要对页码、页眉、章节分页、图题表题位置和公式编号位置等版面规则进行分析，并通过可配置规则适配不同论文规范，从而尽可能覆盖常见论文中的格式问题。
从研究意义来看，本文首先具有较强的实际应用价值。论文写作几乎是每位高校学生都会接触到的任务，而论文格式检查是一项高频且耗时的工作。若能够通过系统自动发现部分明显格式问题，就可以在一定程度上减轻人工校对压力，提高论文修改效率。与此同时，本文也具有一定的技术研究意义。论文格式检查并不是单纯的文本比对问题，而是同时涉及页面版面分析、文本位置理解、文档结构解析和结果定位展示等多个环节。本文以页面表示分析为主要切入点，并辅以Word对象解析完成规则补充，这对于相关文档智能处理任务也具有一定参考意义。
除此之外，本文还具有一定的工程实现意义。本文不仅关注规则本身的判断，还重视系统整体流程的搭建，包括文档输入、规则调度、结果合并、页面定位和图形化展示等内容。通过完成一个可运行的原型系统，可以更直观地验证相关技术方案的可行性。因此，本文的研究既来源于真实需求，也服务于真实需求，同时兼顾了一定的技术探索与工程实现价值。
1.3 国内外研究现状
近年来，随着文档数字化程度不断提高，文档格式自动检查逐渐成为文档处理与智能审校领域的重要研究方向。现有相关研究主要集中在文档版面分析、文档结构理解、文档要素检测与自动审校等方面。
在面向文档版面分析与文档理解的研究中，国内学者已开展了较为系统的探索。叶栩见[1]针对文档版面分析中的边缘信息利用不足和多模态协同问题，提出了基于边缘信息引导与可变形注意力的物理布局分析方法，并设计了基于多模态协同的逻辑结构提取方法，为文档结构化理解和版面要素识别提供了新的思路。赵钰豪等[2]系统回顾了文档解析技术从传统规则方法到深度学习方法的发展过程，重点分析了场景文本检测、表格理解和LayoutLM系列多模态预训练模型，指出多模态预训练模型已成为文档解析的重要发展方向。王家宁等[3]从版面智能布局定义出发，整理归纳了当前基于深度学习的版面智能布局算法，探讨了生成式设计在版面布局中的应用趋势，为版面自动分析与生成提供了方法论支持。刘成林等[4]对文档智能分析与识别技术进行了较全面的综述，从图像预处理、版面分析到文本识别等多个环节总结了深度学习方法的研究进展，并指出了识别精度、自适应性和可解释性等方面仍存在的问题。
在具体版面分析算法方面，相关研究主要围绕检测精度、模型轻量化和复杂场景适应性展开。蔡云冰等[5]提出了面向移动端部署需求的轻量化卷积神经网络文档版面分析算法，在保证较少参数量的同时取得了较好的检测效果。张河锁[6]针对复杂多场景文档版面分析问题，构建并公开了自然拍照手写文档数据集，提出了能够处理多层级分割任务的方法，提升了复杂文档场景下的版面分析能力。蔡云冰等[7]进一步提出基于参数重分配策略的通用文档版面分析方法，通过多尺度特征提取和轻量化结构设计提高了模型的通用性与准确率。Shehzadi等[8]提出了一种半监督文档版面分析框架DocSemi，在仅使用少量标注数据的情况下仍能保持较高检测性能，为低资源场景下的文档分析提供了有效方案。
尹玲等[9]针对文档版面分析效率低和训练成本较高的问题，提出了基于改进YOLOv5s 的文档图像版面分析算法，通过改进特征提取与损失函数设计提升检测效果。袁沛愉等[10]围绕多模态元素解析难题，提出了融合多尺度特征的文档版面分析方法，以增强复杂版面场景下的识别能力。Minouei等[11]研究了将自然场景目标检测器迁移到文档版面分析任务中的可行性，验证了基于目标检测思路处理文档布局问题的潜力。马晓松[12]提出了基于多模态学习的文档版面分析框架，通过融合图像特征、布局特征和关键字特征提高复杂布局下的分析准确率。谢海龙等[13]提出了基于复合注意力机制的文档版面分析算法，通过空间注意力、可变形卷积和通道注意力的结合增强了复杂区域检测能力。Zhou等[14]则针对像素分类方法难以保持区域连续性的问题，提出了基于位置编码和边界框规范的文档版面分析方法。
国外研究同样在文档版面分析和文档理解方面发展较快。Shehzadi等[8]提出的半监督框架为低标注成本场景下的版面分析提供了有效思路。Minouei等[11]验证了目标检测方法在文档版面分析任务中的适用性。Zhou等[14]则从版面区域连续性保持的角度，对传统像素分类思路进行了改进。
在面向实际应用的文档审校与格式检测研究中，也已有部分成果。谭伯乐[15]面向公文格式审核需求，提出了基于大语言模型的文档要素检测和语法纠错方法，并结合专门数据集和训练框架实现了较完整的智能审校流程。这类研究说明，文档格式检查不仅可以停留在传统规则匹配层面，也可以与文档检测、文本理解和智能纠错等技术结合起来，形成更完整的文档智能处理系统。
从整体上来看，国内外相关研究已经为文档格式自动检查打下了比较好的理论基础，并且给予了相应的技术支持，特别是在版面分析、文档理解以及文档要素检测这些方面，取得了比较明显的进展。但是，针对于高校论文规范这一类具体场景，自动化的格式检查系统仍然算是比较少的。现有的方法，或者是偏重于底层的版面分析工作，或者是偏重于通用的文档理解工作，对于论文中同时包含的结构类规则、样式类规则和页面类规则的综合检查，其支持还是不够充分的。鉴于此，本文尝试把Word文档对象分析跟页面分析这两种方法结合起来，去构建一个面向论文格式检查场景的辅助系统，借此来提高对多类型格式问题的覆盖能力，同时让结果展示的直观性得到进一步的提升。
1.4 本文研究内容
围绕论文格式自动检查这一目标，本文的研究内容主要集中在几个相互衔接的方面。首先，本文分析了论文撰写规范中常见的格式要求，并对适合自动检查的规则进行了整理和分类，将其划分为更适合通过Word文档对象分析实现的规则，以及更适合通过页面版面分析实现的规则。其次，本文设计了系统总体架构和处理流程，构建了由页面分析层、文档对象分析层和结果处理展示层组成的系统结构，使页面版面分析与文档结构解析能够在同一系统中协同工作。
在功能实现方面，本文分别完成了文档对象侧规则检查和页面版面侧规则检查。前者主要针对摘要、关键词、目录、标题、引文、参考文献以及字体字号、对齐方式、行距等规则展开，后者则利用Word转PDF后的页面文本与坐标提取结果，对页码、页眉、章节另起页、图题表题位置和公式编号位置等问题进行分析。在此基础上，系统进一步实现了结果合并、问题定位和图形化展示功能，使不同来源的问题结果能够被统一组织，并在界面中以问题列表和页面标注的形式呈现。最后，本文通过样本文档和异常样例对系统进行了测试与分析，总结了当前系统的有效性、局限性和后续可改进方向。通过上述研究内容，本文完成了一个以论文格式检查为目标、并以学校论文规范作为重点场景的原型系统实现。

 第2章 相关技术
2.1 Word文档结构与格式特征
Word文档是当前论文写作中最常见的电子文档形式之一，其内容不仅包含普通文本，还包含段落、标题、页眉、页脚、表格、图片、公式、脚注、参考文献等多种结构元素。与纯文本文件相比，Word文档的一个重要特点在于，它同时保存了内容信息和格式信息。例如，同一段文字不仅有文本本身，还可能关联字体、字号、加粗方式、对齐方式、行距、缩进以及所在样式等属性。
对于论文文档而言，Word文档中的格式特征具有较强的层次性。论文一般是由封面、摘要、目录、正文、参考文献、致谢等多个部分共同组成的，不同部分在标题的形式、字体的字号和排版的方式等方面往往会存在明显区别。比如，摘要标题、正文的一级标题、参考文献的内容还有页码，在字体字号的要求上一般都并不相同。所以，要是想实现自动化的格式检查，那就需要先去理解Word文档内部的结构组织方式以及它的格式特征。
从实现的角度来看，Word文档中的论文格式问题可大致划分为两类。第一类涉及结构与样式属性方面的问题，例如标题层级是否正确、字体字号是否符合要求、参考文献编号顺序是否规范；第二类涉及页面版面布局方面的问题，例如页码是否居中、页眉是否从指定位置起始、图题表题摆放位置是否正确等。前者更适合从Word文档对象本身提取，后者则往往需要结合页面最终呈现结果进行分析。正因如此，本文在系统设计中既使用了Word对象分析，也引入了页面分析链路。
2.2 python-docx文档对象解析技术
python-docx是Python环境里经常选用的一种Word文档处理库，可以对.docx文件当中的段落、样式、表格还有节等对象进行读取以及部分的编辑工作。有关于本文所开展研究的Word文档格式检查任务，python-docx借助它较为便捷的结构化访问特性，成为了文档对象分析模块的一个关键基础。
在运用python-docx时，开发者可以把Word文档看作是由多个对象层次构成的一种结构。比如，文档对象包括了多个段落以及节对象，而每个段落里面又包含了若干个run对象，并且每一个run对象都可以对应到各不相同的字体和字号属性。借助这些对象，系统就能够获取到段落文本、标题样式、对齐方式、行距还有部分的页眉页脚信息，进而可以对摘要、关键词、目录、标题、参考文献等内容开展格式检查工作。
不过，python-docx也存在着一定的局限性。它更为适宜去读取文档的逻辑结构以及样式属性，而对于一些底层的比较复杂的内容，比如页码域代码、文本框当中的内容还有最终页面布局信息，仅仅依靠常规对象接口往往是不能完整地获取到的。因此，在本文的实现当中，除了直接运用python-docx所提供出来的对象属性之外，还结合底层的XML结构，对部分特殊内容开展了补充分析工作，以此来提高页码、页脚和页眉等区域信息提取的准确性。
总的来看，python-docx为论文格式检查提供了一个较为良好的文档对象分析基础，尤其适宜于处理标题、段落、字体字号、对齐方式、引文和参考文献等结构与样式属性类规则。
2.3 PDF页面文本提取与坐标分析方法
对于页码、页眉、图题表题位置和公式编号位置等版面问题，仅通过Word文档对象难以准确判断，因此需要引入页面层面的分析方法。本文选择将Word文档转换为PDF，再从PDF中提取页面文本及其坐标信息，以此作为页面分析的基础。
PDF与Word文档相比，更接近文档最终输出时的页面呈现形式。通过PDF页面文本提取，可以获得每一页的整体文本内容，以及文本块、文本行或文本span在页面中的边界框坐标。这些坐标信息能够反映元素在页面中的实际位置，是判断“是否位于顶部”“是否位于底端居中”“是否位于图下表上”等空间关系的重要依据。
在本文系统中，页面分析阶段主要使用两类信息。第一类是整页文本信息，主要用于页面角色识别，例如判断某一页是否为目录页、摘要页或正文页；第二类是局部文本与坐标信息，主要用于页码候选提取、页眉检测、题注定位以及问题可视化标注。通过同时使用这两类信息，系统能够兼顾整体页面识别与局部版面分析。
因此，PDF页面文本提取与坐标分析方法是本文实现页面规则检查的重要技术基础。
2.4 文档页面版面分析相关方法
文档页面版面分析是指从页面角度研究文档中各类内容的分布方式、空间关系和结构区域。相较于只关注文本内容本身，版面分析更重视文本块、标题、页码、图片、表格和公式等元素在页面中的相对位置。
在论文格式检查任务中，版面分析的意义主要体现在两个方面。其一，某些规则本质上依赖空间关系，例如页码是否位于页面底端居中、图题是否位于图下方、表题是否位于表上方、公式编号是否位于公式右侧行末等。其二，页面整体结构本身也包含较强语义信息，例如目录页通常包含点线连接的目录项，正文首页通常具有“第1章”或“绪论”等起始标记，参考文献页和致谢页往往在页面顶部出现明显标题。
本文系统中的页面版面分析方法主要采用基于规则和特征匹配的思路，而不是复杂的深度学习模型。一方面，这是因为当前课题更关注具体规范场景下的可实现性；另一方面，论文的页面结构通常具有较强规律性，利用文本模式、坐标区域和相对位置关系，已经能够支持多数常见版面规则检查。虽然这种方法在复杂文档场景下仍有局限，但对于当前论文格式检查任务具有较好的适用性。
2.5 PyQt 图形界面开发技术
为了使检查结果更直观地呈现给用户，本文采用PyQt构建图形用户界面。PyQt是Python环境下常用的桌面界面开发工具，提供了丰富的窗口控件、布局管理机制和事件响应能力，适合实现具有一定交互复杂度的桌面应用程序。
在本系统中，PyQt主要用于完成以下几类功能。第一，构建文档选择、规则执行和结果展示等基础交互界面；第二，显示PDF页面图像及问题标注结果；第三，实现问题列表与页面跳转之间的联动；第四，支持页面缩放、浏览和结果概览展示。
引入图形界面的意义在于，论文格式检查既要发现问题，还要涉及问题在哪里、是否便于确认。如果系统只输出文字结果，用户仍需要手动翻找文档对应位置，使用体验会受到较大影响。通过PyQt提供的界面展示能力，本文将检查结果、问题列表和页面可视化标注整合到同一界面中，提高了系统的可用性。

 第3章 系统需求分析与总体设计
3.1 系统建设目标
本课题面向论文格式检查场景，目标是设计并实现一个能够对Word文档进行自动检查的软件系统。系统应当可以依据毕业论文、课程论文或其他学术论文的撰写规范，对论文里面的封面、摘要、目录、正文、图表、公式、参考文献等部分开展格式方面的检查工作，并输出相应的错误提示或警告信息，帮助用户找出排版中存在的问题。
传统的Word文档格式检查工作大多依靠文档对象本身的结构信息，比如段落、样式、字体、字号、对齐方式等。这类方法对于文本内容以及样式属性类问题，有较好的检查效果，但是在涉及页面布局以及视觉呈现方面的规则时，往往就会存在一定的局限。比如，页码是否处在页面底端居中、页眉是否从第1章开始启用了、章节标题是否另起一页、图题和表题的位置是否正确等等问题，仅仅依靠文档对象的属性是很难准确判断的。因此，本系统将页面分析作为主要检查路径，并使用文档对象分析对结构和样式规则进行补充。
结合课题要求与实际实现情况，系统建设目标可以概括为几个彼此关联的方面。首先，系统需要完成对Word文档主要结构内容的自动识别与解析，并在此基础上实现文档对象属性类规则的自动检查。其次，系统还应具备对页面布局类规则的检测能力，以弥补仅依赖文档对象分析时在版面判断上的不足。在结果层面，系统需要将.docx检查结果与页面分析结果统一整合，并进一步提供问题定位与可视化展示能力。通过这些目标的实现，构建一个兼顾结构信息与页面信息论文格式检查系统，提高论文格式检查的自动化程度和结果的直观性。
3.2 系统功能需求分析
根据论文格式检查任务的特点，系统需要具备文档输入、格式检查、结果汇总和可视化展示等基本功能。结合当前系统实现情况，可以将功能需求划分为以下几个方面。
3.2.1 文档输入与预处理需求
系统首先应支持用户选择待检查的Word文档作为输入对象。由于部分页面规则无法直接依赖Word文档对象完成判断，因此系统还需要具备将Word文档转换为PDF页面表示的能力，或者支持用户直接选择已有的同名PDF文件参与页面分析。
在文档输入完成后，系统应分别完成两类预处理：一类是对.docx文档进行结构解析，提取段落、标题、样式等对象信息；另一类是对PDF页面进行文本与坐标提取，为后续页面规则检查和问题标注提供基础数据。
3.2.2 文档对象检查需求
文档对象侧主要面向内容结构与格式属性规则。系统应能够对论文中的主要板块进行检查，包括封面、中文摘要、英文摘要、目录、正文、参考文献和致谢等部分。同时，还应能够进一步检查字体、字号、行距、对齐方式、字符间距、标题编号、关键词数量、摘要长度、参考文献编号顺序、引文上标等具体规则。
这部分功能的重点在于利用python-docx获取到的结构化信息，对论文内容本身的规范性进行判断。
3.2.3 页面分析检查需求
页面分析侧主要面向页面布局与视觉呈现规则。系统应能够识别PDF页面中的文本块及其坐标位置，并在此基础上完成以下类型的检查：页码位置与连续性检查、页眉启用边界检查、目录页识别、章节是否另起一页、图题表题相对位置检查、公式编号位置检查以及注释位置辅助检查等。
这部分功能的重点在于结合页面中的文本分布、坐标信息和局部区域特征，对论文的版面格式进行分析。
3.2.4 检查结果展示需求
系统不仅要输出检查结果，还要尽量让结果便于理解。为此，系统需要提供统一的问题列表展示能力，能够对不同来源的检查结果进行汇总，并按照一定顺序编号显示。同时，系统还应支持错误位置的页面可视化标注，使用户能够在页面图像中直接看到问题所在区域，提高结果的可读性和实用性。
3.2.5 用户交互需求
在用户交互方面，系统应支持文档选择、执行检查、问题浏览、页面跳转和页面缩放等基础操作。用户点击问题列表中的某一项后，系统应能够切换到对应页面并在左侧显示相关标注，便于快速定位问题。

 第4章 系统实现
4.1 文档对象分析实现
文档对象分析是整个系统的起点。程序在接收到用户选择的.docx文件后，先通过DocxContextBuilder构建规则上下文，再由DocxStructureAdapter和python-docx对文档内容进行解析。这个过程不是简单地把段落文本读出来，而是要把后续检查需要用到的结构、样式和板块信息整理成统一的数据对象，供规则层反复调用。
从实际代码流程来看，这一阶段主要完成三件事。第一件事是识别文档由哪些部分组成，包括封面、摘要、目录、正文、参考文献、致谢等。第二件事是提取各部分中的段落内容、标题层级、运行块文本、表格和节对象信息。第三件事是把解析结果写入RuleContext，让文档对象规则和混合检查流程都能直接使用。这样处理之后，规则实现不需要再次遍历全文，系统的整体结构也会更清晰。
4.1.1 文档上下文的构建
在代码实现中，HybridProcessor.process()并不会直接调用具体规则，而是先执行 _build_context()。该方法内部先通过self.docx_builder.build(file_path)构建文档对象上下文，再根据是否存在同名PDF或用户指定PDF，继续准备页面侧数据。这样设计的好处是，文档对象分析与页面分析能够共用同一份上下文信息，避免不同模块分别解析同一个文件。
DocxContextBuilder的职责是把原始解析结果包装成RuleContext。如果文档解析成功，就把docx_obj、docx_sections和文档部分顺序信息写入上下文；如果解析失败，则把异常信息写入extras["docx_parse_error"]。后续规则在执行前会先判断这个状态，从而避免解析失败后继续进入检查逻辑。这种处理方式虽然不复杂，但对系统稳定性比较重要，尤其是在样本文档格式不统一时，可以避免整条检查链路直接中断。
4.1.2 文档结构与格式信息提取
在文档对象分析阶段，系统首先要解决“文档有哪些部分”的问题。当前实现会结合标题文本、板块关键词和段落顺序，将文档划分成摘要、目录、正文、参考文献等部分，并把各部分的段落内容分别保存。对于正文部分，还会继续提取标题列表、引文标识、参考文献条目、图题表题等信息。这样做的目的，是让后续规则不再面对一篇未分类的长文档，而是直接面向已经整理好的结构化数据。
除了结构划分外，系统还会提取与格式相关的属性。例如段落的对齐方式、行距、首行缩进、段前段后间距，运行块的字体、字号、上下标状态，以及节对象中的页边距、页眉页脚距离等。以页码、页眉和页脚相关规则为例，它们在stage2_rules.py中并不是只读取普通段落，而是会进一步访问节对象、页脚节点和XML信息。这样才能处理Word中常见的自动页码、域代码和链接页眉页脚等情况。
4.1.3 样式继承与特殊对象处理
文档对象分析里最容易出问题的部分，其实不是读取文本，而是还原真实格式。因为Word中很多样式并不直接写在当前run上，而是来自段落样式、字符样式，甚至来自文档默认样式。如果只读取当前节点的显式属性，就很容易把样式继承漏掉。项目前期出现过英文标题Abstract被误报、正文中python-docx和Open XML被误报为“字体不是宋体”的情况，本质上都是这里处理得不够细。
针对这个问题，当前实现补充了样式继承回溯和字体候选项读取逻辑，并在字体检查时区分中文run和纯西文run。也就是说，当规则要求是中文字体时，纯英文内容不会再被直接拿去和“宋体”“黑体”做硬匹配。这一改动对减少误报很有帮助，也让文档对象规则更符合真实排版习惯。
对于参考文献自动编号、页码域和页眉页脚这类对象，系统还继续下探到底层XML。以PageNumberFormatRule为例，它并不是只检查“是否存在数字”，而是会调用_footer_page_number_nodes()读取页脚中真正的页码节点，再结合字体、字号、对齐方式进行判断。HeaderFormatRule和FooterFormatRule也采用了类似做法，用来处理不同节之间可能存在的页眉页脚继承关系。这样一来，文档对象分析不仅能支持普通文本规则，也能覆盖 Word 中较难直接读取的对象。表4-1列出了这一阶段整理出的主要中间数据项。

4.2 页面分析与角色识别实现
页面分析模块是本系统格式检查流程中的重要组成部分。系统进一步将文档转换为PDF，从页面角度分析各部分内容在版面中的实际分布情况。当前实现主要依赖PDF提取出的文本和坐标信息，对页面中的标题、页码、页眉页脚、图题、表题等对象进行定位，并结合它们的相对位置判断相关格式是否符合要求。这种处理方式与python-docx侧的样式属性检查形成了互补，使系统既能检查文档结构和格式属性，也能检查页面排版效果。
4.2.1 页面对象的构建
在混合检查流程中，如果系统能够找到用户指定的PDF，或者在Word同目录下找到同名PDF，就会进入页面分析阶段；否则会先执行Word转PDF。对应的代码位于 MainWindow.ensure_pdf_for_hybrid_check()和HybridProcessor._build_context()。前者负责在界面层准备PDF输入，后者负责调用PdfExtractor.extract()将PDF解析为页面对象列表。
每一个页面对象中都包含页号、页面宽高、整页文本、文本span以及每个span的坐标框。这样做之后，页面侧规则就不需要直接操作原始 PDF 文件，而是只面对统一的数据结构。对后续的页码检查、页眉检查、题注检查和错误标注来说，这个页面对象都是共同的数据基础。
4.2.2 页面角色识别
页面分析的第一步不是立即检查格式，而是先判断当前页面属于哪一部分。当前项目在page_roles.py中定义了PAGE_ROLE_CN_ABSTRACT、PAGE_ROLE_EN_ABSTRACT、PAGE_ROLE_CATALOGUE、PAGE_ROLE_MAIN、PAGE_ROLE_BACKMATTER和PAGE_ROLE_OTHER六类页面角色，并通过build_page_roles()对整份PDF逐页标注。
这一过程主要依靠页面顶部标题、目录点线项、章节标题和正文后置部分标题进行识别。例如，如果页面中出现“目录”标题，且存在较多点线连接项和编号条目，就优先判为目录页；如果页面顶部更像“第1章”“第一章”“绪论”等正文起始形式，就把这一页及其后续页面纳入正文范围；如果页面顶部出现“参考文献”“致谢”或“附录”等标题，则把该页之后的内容划入正文后置部分。中文摘要与英文摘要则主要利用顶部标题“摘要”和“Abstract”来区分。
这个角色识别结果不仅会被页码和页眉规则复用，在GUI的错误定位中同样会用到。也就是说，系统后面无论是检查页码连续性，还是把“摘要错误”限制在摘要页范围内，本质上都依赖这一阶段的判断。因此页面角色识别虽然属于预处理步骤，但它对误报率影响很大。项目中出现过目录里的“第4章”被误识别成页码的问题，后续也是在这里不断收紧规则后才逐渐修正。
4.2.3 页面格式规则的实现
在页面角色确定之后，系统开始执行页面侧规则。与文档对象规则不同，页面规则更关注坐标、区域和相对位置。例如PageNumberPresencePdfRule会先调用_best_page_number_candidate()在页面底端寻找页码候选，再结合页面角色判断这一页是否本来就应该有页码；PageNumberBottomCenterPdfRule则进一步判断页码候选是否接近页面底端居中位置。
页眉检查的做法与此类似。HeaderTopContentPdfRule会提取页面顶部文本，判断是否识别到规范页眉内容；HeaderStartBoundaryPdfRule则会结合正文起始页，判断页眉是否从正文开始才出现。换句话说，页面规则不是单看某一页，而是会结合这页在整篇文档中的角色来判断它是否合理。
除了页码和页眉外，图题表题、章节另起页和公式编号位置等规则也依赖页面空间关系。以公式编号为例，规则并不是只检查“有没有(3-1)这样的编号”，而是还要判断编号是否位于公式行右侧末端；图题表题规则也不是只看是否出现“图”或“表”字样，而是要结合其与图表区域的上下关系进行判断。正因为这些规则面向的是页面最终呈现效果，所以它们更能体现本课题中“页面分析参与格式检查”的特点。表4-2汇总了页面规则的主要检测对象及判断依据。

4.3 规则调度与典型规则实现
这一节对应的是系统真正执行检查的主链路。前面两节分别解决了“Word文档中有哪些对象”和“PDF页面上有哪些版面信息”两个问题，而在实际运行时，还需要有一个中间层把这两类信息组织起来，再交给不同规则分别检查。这个工作主要由model/hybrid/pipeline.py中的HybridProcessor完成。用户在界面中点击“文档检查”后，MainWindow.hybrid_check()会先读取当前格式设置，再调用ensure_pdf_for_hybrid_check()准备PDF文件，最后把文档路径、PDF路径和用户参数一并传入HybridProcessor.process()。从这一刻开始，系统就进入统一的混合检查流程。
4.3.1 混合检查流程的组织
HybridProcessor.process()的实现可以看成一条比较清晰的串行流程。程序首先调用 _build_context()构建RuleContext，其中既包含DocxContextBuilder.build()得到的文档对象信息，也会在PDF存在时通过PdfExtractor.extract()把页面文本、坐标框和页尺寸一起写入上下文。这样做的目的，是让后续规则不必再各自读取文件，而是都从同一个上下文对象中取数。结合项目实际来看，这种写法比较合适，因为摘要字数、关键词数量、标题字体等规则需要访问段落和run，而页码位置、页眉内容、图题表题相对位置等规则又必须依赖页面坐标；如果没有统一上下文，后续规则很容易重复解析文件，也不利于结果合并。
上下文构建完成后，process()会通过router.select()取得当前需要执行的规则集合，再按照docx和pdf两个引擎分别循环调用rule.check(ctx)。这里并没有把所有规则混在一起顺序执行，而是先按照引擎分类，再把每个规则产生的Issue收集到grouped_issues中。这样处理有两个好处。一是便于保留规则来源，后面在界面中可以区分问题是来自文档对象检查还是页面分析检查；二是如果某条规则运行出错，程序会在except分支中构造一条执行失败的问题对象，而不会导致整次检查中断。对于毕业设计项目来说，这一点很重要，因为规则数量已经比较多，单条规则的健壮性不应该影响整条链路的可用性。
4.3.2 文档对象规则与页面规则的分工
从当前项目的代码来看，两类规则的职责划分已经比较明确。create_default_registry()中注册的ChineseAbstractLengthRule、KeywordCountRule、CitationSuperscriptRule、ReferenceEntryFormatRule、ReferenceNumberSequenceRule、FontSizeFormatRule等规则，主要依赖python-docx提取出的结构和样式信息，适合放在文档对象侧完成。这类规则关注的是“内容属于哪一部分”“某段文字的字体字号是什么”“参考文献序号是否连续”这一类结构化问题，单靠页面外观不容易判断准确。
另一部分规则则更适合放在页面分析侧。例如PageNumberPresencePdfRule、PageNumberBottomCenterPdfRule、HeaderTopContentPdfRule、HeaderStartBoundaryPdfRule会重点使用页码候选框、页面顶部文本和页面角色信息；图题、表题、公式编号和目录展示相关规则则会使用文本块的上下关系、左右位置以及页面中的相对坐标。以这次项目中反复调整的页码检测为例，仅靠Word对象很难判断数字是否真的位于页脚中部，也无法区分目录中的“第 4 章”和真正的页码数字，因此必须借助page_roles.py中的页面角色判断和PDF文本坐标分析。换句话说，文档对象侧解决的是“文档写了什么”，页面分析侧解决的是“这些内容最后排成了什么样”，两者缺一不可。
4.3.3 统一问题对象的构建与输出
各条规则执行结束后，HybridProcessor会把结果交给IssueMerger.merge()统一合并，再通过issue_to_dict()转换成界面可以直接使用的字典结构。这个字典除了标题、描述、严重程度和规则编号以外，还会尽量保留page、bbox、metadata等字段。前两个字段直接服务于页面标注，metadata则会继续保存板块名称、原始文本、引文marker、参考文献条目等附加信息，供后面的定位函数使用。最后build_summary()会统计问题总数和不同严重程度的数量，build_context_status()会记录本次是否成功使用了PDF页面上下文。这样处理后，界面层不需要了解每条规则的内部实现，只需要接收统一结果并负责展示即可。图4-1展示了这条从上下文构建、规则分流到结果回收的主流程。

4.4 问题定位与可视化实现
这一节更偏向系统实现层面的展示。对用户来说，最容易直接感受到的不是规则类名，而是“检查结果能不能看懂”“点了某条问题之后能不能快速找到页面上的位置”。因此本项目没有把结果只停留在文本列表上，而是在gui/main_window.py中补充了问题定位、页面跳转和矩形标注三部分逻辑。整个界面基于PyQt6实现，左侧用于显示PDF页面的可视化结果，右侧用于显示检查概览、问题列表和参数设置，界面中的按钮、列表和页码控件则通过信号与槽机制组织起来。
4.4.1 文件导入与检查触发
在交互入口上，主窗口提供了select_file()和select_pdf()两个方法，前者通过QFileDialog选择Word文档，后者用于手动补充PDF路径。考虑到实际使用时用户并不一定会提前准备好PDF，因此我在界面中增加了ensure_pdf_for_hybrid_check()。该方法会优先使用用户手动指定的PDF；如果没有，就调用docx_to_pdf_converter.convert()自动把Word转成同名PDF，并把生成路径重新写回界面。这样处理后，用户点击一次“文档检查”即可完成文档准备和规则执行，不需要再手动切换软件导出PDF。
真正开始检查时，hybrid_check()会先调用get_format_settings()读取右侧各个QComboBox当前选中的字体、字号和行距参数，再把这些参数作为user_formats传入HybridProcessor.process()。如果PDF转换失败，程序会在RuntimeError分支里弹出QMessageBox提示，并自动退回到只执行文档对象检查的模式。这样写的好处是主流程仍然可用，不会因为某次转换失败就完全无法给出检查结果。
4.4.2 检查结果列表与页面联动
混合检查完成后，display_hybrid_results()会先清空旧结果，再把新返回的问题对象按严重程度和显示顺序重新整理，然后分别放入“检查概览”和“问题列表”两个区域。列表中的每一项并不是简单文本，而是由create_issue_item_widget()构造的单独组件，里面会显示序号、板块名称和简短问题说明。这样做的目的，是让每条问题都成为一个可点击的交互单元，后面可以直接与左侧页面联动。
当用户点击右侧某一条问题时，程序会调用navigate_to_issue()。这个方法会先检查当前问题是否已经带有明确页码；如果有，就直接把左侧页码控件page_spin跳到目标页。如果问题本身没有页码，但在可视化阶段已经补出了display_index对应的页面位置，也会优先使用补定位后的结果。与此同时，populate_current_page_issue_buttons()会把当前页涉及的问题同步列到左下方的小列表中，用户可以在页面内部继续切换同一页的多个问题。这种设计虽然不复杂，但实际使用时比只显示一长串报错文本清楚得多。
4.4.3 文档对象问题的近似定位
页面规则一般会直接输出page和bbox，因此比较容易标注；真正麻烦的是文档对象规则。以摘要长度、关键词格式、引文上标和参考文献内容这类问题为例，它们本身是在Word结构上判断出来的，规则执行时通常拿不到PDF坐标，因此界面层还需要做一次“从docx问题回找PDF位置”的近似定位。这部分主要集中在collect_visual_issues()、locate_docx_issue_bbox()、locate_docx_marker_bbox()和locate_reference_entry_boxes()这几个方法中。
具体实现时，我先利用build_pdf_page_roles()为每页分配角色，再通过allowed_roles_for_issue()和iter_issue_candidate_pages()缩小搜索范围。比如中文摘要相关问题优先在中文摘要页查找，目录问题限制在目录页，参考文献问题主要放在PAGE_ROLE_BACKMATTER对应的后置部分，正文中的引文上标则优先放在正文页处理。这样做的原因很直接，因为如果不先按板块过滤，像“摘要”“关键词”这样的短文本很容易在封面、目录或正文中反复误命中，前面调试时出现过的定位漂移问题，大多都和候选页过宽有关。
在候选页确定后，程序会调用group_page_lines()把同一水平带内的span合并成文本行，再通过build_line_windows()或iter_page_windows()生成连续1至4行的窗口，参考文献条目则放宽到6行。locate_docx_issue_bbox()会把问题文本归一化后，与这些窗口逐个比较，并综合考虑命中长度、额外字符数量、起始位置以及窗口行数给出一个匹配分数，最后选取得分最高的区域作为标注框。相较于最开始只按单行匹配的方式，这种多行窗口更适合处理正文长段落、换行标题和跨行参考文献条目。
对于局部标记更明显的问题，程序又单独写了两个分支。locate_docx_marker_bbox() 主要处理引文上标类问题，它会优先读取metadata中的marker，例如[10]、[11]，然后直接到正文页的文本行里搜索这个标记，再把marker所在行整体框出。locate_reference_entry_boxes()则用于参考文献内容问题，它会先按|拆开多个条目，再优先用诸如[17]这样的编号去匹配后置部分的窗口。这样做虽然仍然属于近似定位，但比拿整段参考文献内容去全文搜索稳定得多，也更符合项目当前的实际效果。
4.4.4 错误标注框的绘制与缩放
定位结果确定后，左侧页面的绘制工作由render_pdf_page_with_issue_boxes()完成。该方法使用fitz打开PDF，并通过page.get_pixmap()把目标页渲染成位图，再借助QImage和QPixmap转换成PyQt可以显示的图像对象。接着，程序使用QPainter、QPen和QColor在页面上绘制矩形框，并把“问题序号+板块名称”写到框的上方。为了让严重错误更醒目，代码中还根据severity区分了边框颜色和线宽。
缩放功能并不是重新渲染整份PDF，而是由refresh_visualization_pixmap()根据当前显示模式重新缩放已经绘制好的QPixmap。当用户选择“适应宽度”时，程序会读取滚动区域 image_scroll.viewport().width()，按照当前可视宽度进行等比缩放；当用户切换为百分比模式时，则使用display_zoom_factor对图片宽度进行重新计算。这样处理后，缩放操作只影响显示层，不会重复执行规则或重新搜索定位位置，界面响应会更加直接。结合update_issue_visualization_page()和scroll_to_issue()的联动逻辑，最终就形成了“右侧列表点击一条问题，左侧自动跳页并显示标注框，当前页问题列表同步更新”的完整可视化链路。图4-4所示界面可以较直观地反映这一实现效果。

5.5 综合检查结果分析
从系统整体运行的效果来看，把页面分析和文档对象分析结合在一起，能比只用单一方式覆盖到更多的格式检查需求。一方面，页面分析侧的规则更适合去处理页码、页眉、章节分页、图题表题和公式编号这些版面问题；另一方面，文档对象侧的规则在字体、字号、标题编号、引文和参考文献等结构与样式问题上更擅长，两边的功能有很强的互补性。
在结果展示上，系统能把不同来源的问题统一汇总到同一个结果列表里，并给出检查概览、问题条目和页面标注信息，和只输出一段段文字描述的做法比起来，这种混合展示方式会让用户更容易理解问题到底出在哪里。比如，当用户看到“页码可能未位于页面底端居中”这样的提示后，可以直接在页面预览里查看对应的位置，很快就能判断出这个问题是不是真的存在了。
同时，混合检查也反映出了不同来源规则在准确度上的差别。文档对象规则检查结构、样式属性时一般比较稳定，可一旦碰到复杂的底层对象，就有可能出现误报；页面规则判断版面时更加直观，但在目录页、复杂文本块布局这类场景下，还是容易受到一些干扰。所以，当前系统更适合作为一个辅助检查工具，主要作用就是帮用户把那些高概率的问题快速找出来，再由人工去完成最终确认。图5-5展示了多格式问题文档的总检查结果；图5-6展示了无格式问题文档的检查结果，同时系统会提示所有检查链路都已正常执行。

 第6章 结论与展望
6.1 论文工作总结
本文围绕论文格式自动检查需求，完成了一个基于Python的Word文档格式检查系统原型。围绕高校论文撰写规范，系统建立了从文档输入、规则检查到结果展示的完整处理流程，并重点面向论文中常见的结构、样式和页面版面问题开展自动分析。
在研究与实现过程中，本文完成了规则需求梳理、系统架构设计、关键功能开发以及测试分析等工作。其中，系统以页面分析为主要路径，对页码、页眉、章节分页、图题表题位置和公式编号位置等问题进行判断，并结合Word文档对象分析补充摘要、关键词、目录、标题、参考文献和引文等结构与样式类检查。通过引入问题定位与图形界面展示功能，检查结果能够以更直观的形式反馈给用户。整体来看，本文较好地完成了课题预期目标。
6.2 主要成果总结
本文的主要成果体现在三个方面：一是，完成了Word文档格式检查系统的总体设计与开发，让整个软件的处理流程变得比较完整了；二是，构建了一套页面分析加文档对象分析相结合的检查机制，这样一来，系统既能处理摘要、关键词、标题、引文和参考文献这类结构与样式上的问题，也能照顾到页码、页眉、章节分页、图题表题位置和公式编号位置这些版面问题；三是，实现了问题定位与可视化展示的功能，不同来源的检查结果能在同一个界面上统一呈现出来，用户也可以很方便地找到问题所在的位置。在测试和调试的过程里，我们还专门针对页码字号识别、目录页数字被误认成页码，还有检查结果的展示逻辑等一系列问题，持续地做了优化，系统的整体稳定性也因此有了一些提升。总的来看，本文实现的系统已经具备了比较完整的论文格式辅助检查能力，能给用户带来很有实际价值的自动化支持。
6.3 不足与后续展望
虽然本文已经完成了系统主要的设计与实现工作，但受时间、经验和实现条件的限制，目前的系统还是留下了不少可以改进的地方。其中一点是，系统里有部分规则采用了启发式的实现方式，一旦碰到的文档排版形式比较复杂，或者跟常规论文样式差别很大，就容易出现误报或漏报的情况；后续可以多收集一些样本文档，对规则条件做更细致的细化和优化，让系统在复杂场景下的稳定性变得更好一些。
另一方面，文档对象分析在处理Word底层复杂结构时也存在一定局限。像页码、页眉、页脚等区域，经常会涉及域代码、文本框以及样式继承关系，这些情况都会影响系统对格式属性的准确提取。后续如果能够进一步加强对底层XML结构的分析能力，规则判断的可靠性还可以继续提高。与此同时，当前页面分析链路主要依赖PDF文本与坐标提取，因此在面对扫描版文档、文本提取质量较差的PDF，或者版式更复杂的页面时，分析效果仍会受到一定限制。后续可以继续围绕复杂页面适应能力、版面定位精度和规则鲁棒性等方面开展优化。
另外，现在系统的问题定位主要还是近似定位，特别是文档对象侧的问题在页面中的位置匹配，有时会存在一些误差，往后可以从文本匹配策略、多区域候选排序，还有跨页上下文分析这些方面去改进，把问题标注的精确度再往上提一提。由于目前的系统主要是照着单一学校的论文规范去实现的，如果后面能把规则配置进一步参数化和模板化，就有希望推广到更多学校和更多文档规范场景中，提升系统的通用性与推广价值。总的来看，本文实现的这个系统，已经给论文格式自动检查打了一个可运行的基础，未来要是在规则精度、页面识别能力、问题定位精度和规范适配范围等方面继续完善下去，它的应用价值一定还能再上一个台阶。
