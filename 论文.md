# 论文框架（草案）

## 摘要

毕业论文格式检查是论文写作过程中的重要环节，但传统人工校对方式存在工作量大、效率低和易遗漏问题等不足。针对这一问题，本文设计并实现了一个基于 Python 的 Word 文档格式检查系统，用于辅助用户发现毕业论文中的常见格式错误。

本文首先结合毕业论文撰写规范，对论文中的格式规则进行分析，并将其划分为文档对象属性类规则和页面版面类规则两类。针对前一类规则，系统基于 Word 文档对象解析技术，对摘要、关键词、目录、标题、引文、参考文献以及字体、字号、对齐方式和行距等内容进行检查；针对后一类规则，系统引入页面分析链路，将 Word 文档转换为 PDF 后提取页面文本及坐标信息，对页码、页眉、章节分页、图题表题位置和公式编号位置等问题进行判断。在此基础上，系统实现了混合规则调度、结果合并、问题定位和图形界面展示功能，使用户能够在查看问题描述的同时观察问题在页面中的大致位置。

测试结果表明，本文实现的系统能够较好地完成毕业论文中多数常见格式问题的自动检查，尤其在文档结构检查、样式属性检查、页码页眉检查和结果可视化展示方面具有较好的实用效果。同时，系统在目录页数字识别、复杂页脚结构处理和文档对象问题精确定位等方面仍存在一定不足，后续仍有进一步优化空间。

总体来看，本文完成了一个面向毕业论文格式检查场景的原型系统实现，为毕业论文排版规范化提供了一种具有实际应用价值的辅助工具。

## Abstract

Format checking is an important part of thesis writing, but traditional manual proofreading is often time-consuming, inefficient, and prone to omission. To address this problem, this paper designs and implements a Python-based Word document format checking system to assist users in identifying common formatting errors in undergraduate theses.

First, the thesis formatting rules are analyzed and divided into two categories: document object attribute rules and page layout rules. For the first category, the system uses Word document object parsing technology to check abstracts, keywords, tables of contents, headings, citations, references, as well as fonts, font sizes, alignment, and line spacing. For the second category, the system introduces a page analysis pipeline. After converting a Word document into PDF, the system extracts page text and coordinate information to analyze page numbers, headers, chapter pagination, caption positions, and formula numbering positions. On this basis, the system further implements hybrid rule scheduling, result merging, issue localization, and graphical interface presentation, so that users can not only read issue descriptions but also view their approximate positions on the page.

The test results show that the system can effectively detect most common formatting problems in undergraduate theses, especially in document structure checking, style attribute checking, page number and header checking, and visual result presentation. At the same time, there are still some limitations in directory number recognition, complex footer structure handling, and precise localization of document-object-based issues, which need further improvement.

Overall, this paper completes the implementation of a prototype system for thesis format checking and provides a practical auxiliary tool for standardizing thesis layout.

### 1.1 课题研究背景

随着高校毕业设计（论文）写作逐渐全面电子化，论文排版与格式规范问题也变得越来越突出。对于本科毕业论文而言，除研究内容本身外，论文格式是否符合学校统一要求，同样是论文质量评价的重要组成部分。毕业论文通常包含封面、摘要、目录、正文、参考文献、附录、致谢等多个部分，不同部分在字体、字号、对齐方式、行距、页眉、页码以及图表公式编号等方面均有较为明确的规范要求。如果这些内容依靠人工逐项核对，不仅工作量较大，而且容易因为疏忽而遗漏问题。

从实际写作过程来看，学生在论文撰写中经常会遇到格式设置不统一、目录与正文不一致、页码位置不正确、标题层级错误、图题表题位置不规范、参考文献编号混乱等问题。特别是在论文多次修改和反复排版后，原本已经调整好的格式可能再次发生变化，这使得后期人工校对的负担进一步增加。因此，研究一种能够自动发现论文格式问题的辅助检查系统，具有较强的现实意义。

另一方面，近年来文档处理、智能排版分析和文本信息提取技术不断发展，为论文格式自动检查提供了技术基础。利用 Word 文档对象解析技术，可以较方便地获取段落、标题、字体和样式等结构化信息；利用 PDF 页面分析技术，可以进一步获取页面中的文本位置和版面布局特征。将这两类信息结合起来，有望实现对论文结构类规则和版面类规则的联合检查，从而提高格式检查的覆盖范围和准确性。

基于上述背景，本文选择毕业论文格式自动检查作为研究对象，设计并实现一个基于 Python 的 Word 文档格式检查系统，以期在毕业论文排版规范化方面提供一种可行的辅助工具。

### 1.3 国内外研究现状

近年来，随着文档数字化程度不断提高，文档格式自动检查逐渐成为文档处理与智能审校领域的重要研究方向。现有相关研究主要集中在文档版面分析、文档结构理解、文档要素检测与自动审校等方面。

在面向文档版面分析与文档理解的研究中，国内学者已开展了较为系统的探索。叶栩见[1]针对文档版面分析中的边缘信息利用不足和多模态协同问题，提出了基于边缘信息引导与可变形注意力的物理布局分析方法，并设计了基于多模态协同的逻辑结构提取方法，为文档结构化理解和版面要素识别提供了新的思路。赵钰豪等[4]系统回顾了文档解析技术从传统规则方法到深度学习方法的发展过程，重点分析了场景文本检测、表格理解和 LayoutLM 系列多模态预训练模型，指出多模态预训练模型已成为文档解析的重要发展方向。刘成林等[6]对文档智能分析与识别技术进行了较全面的综述，从图像预处理、版面分析到文本识别等多个环节总结了深度学习方法的研究进展，并指出了识别精度、自适应性和可解释性等方面仍存在的问题。

在具体版面分析算法方面，相关研究主要围绕检测精度、模型轻量化和复杂场景适应性展开。蔡云冰等[3]提出了面向移动端部署需求的轻量化卷积神经网络文档版面分析算法，在保证较少参数量的同时取得了较好的检测效果。张河锁[7]针对复杂多场景文档版面分析问题，构建并公开了自然拍照手写文档数据集，提出了能够处理多层级分割任务的方法，提升了复杂文档场景下的版面分析能力。蔡云冰等[8]进一步提出基于参数重分配策略的通用文档版面分析方法，通过多尺度特征提取和轻量化结构设计提高了模型的通用性与准确率。尹玲等[10]、袁沛愉等[11]、谢海龙等[14]则分别从改进 YOLO、跨层多尺度特征融合和复合注意力机制等角度对文档版面分析算法进行了优化，在不同公开数据集上取得了较好的结果。

国外研究同样在文档版面分析和文档理解方面发展较快。Shehzadi 等[9]提出了一种半监督文档版面分析框架 DocSemi，在仅使用少量标注数据的情况下仍能保持较高检测性能，为低资源场景下的文档分析提供了有效方案。Minouei 等[12]研究了将自然场景目标检测器应用于文档版面分析的可行性，验证了基于目标检测思路处理文档布局问题的潜力。Zhou 等[15]针对像素分类方法难以保持分析区域连续性的问题，提出了基于位置编码和边界框规范的文档版面分析方法，提高了版面区域识别的连续性与准确性。

在面向实际应用的文档审校与格式检测研究中，也已有部分成果。谭伯乐[2]面向公文格式审核需求，提出了基于大语言模型的文档要素检测和语法纠错方法，并结合专门数据集和训练框架实现了较完整的智能审校流程。这类研究说明，文档格式检查不仅可以停留在传统规则匹配层面，也可以与文档检测、文本理解和智能纠错等技术结合起来，形成更完整的文档智能处理系统。

总体来看，国内外研究已经为文档格式自动检查提供了较好的理论基础和技术支持，尤其是在版面分析、文档理解和文档要素检测方面取得了明显进展。然而，面向高校毕业论文这一特定场景的自动化格式检查系统仍相对较少。现有方法要么偏重底层版面分析，要么偏重通用文档理解，对于毕业论文中同时包含结构类规则、样式类规则和页面类规则的综合检查支持仍不够充分。因此，本文尝试结合 Word 文档对象分析与页面分析方法，构建一个面向毕业论文格式检查场景的辅助系统，以提高对多类型格式问题的覆盖能力和结果展示的直观性。

## 第2章 相关技术与理论基础

### 2.1 Word文档结构与格式特征

Word 文档是当前论文写作中最常见的电子文档形式之一，其内容不仅包含普通文本，还包含段落、标题、页眉、页脚、表格、图片、公式、脚注、参考文献等多种结构元素。与纯文本文件相比，Word 文档的一个重要特点在于，它同时保存了内容信息和格式信息。例如，同一段文字不仅有文本本身，还可能关联字体、字号、加粗方式、对齐方式、行距、缩进以及所在样式等属性。

对于毕业论文而言，Word 文档中的格式特征具有较强的层次性。论文一般由封面、摘要、目录、正文、参考文献、附录、致谢等多个部分组成，不同部分在标题形式、字体字号和排版方式上往往存在明显区别。例如，摘要标题、正文一级标题、参考文献内容和页码在字体字号要求上通常并不相同。因此，若要实现自动化格式检查，首先需要理解 Word 文档内部的结构组织方式及其格式特征。

从实现角度看，Word 文档中的论文格式问题大致可以分为两类。第一类是结构与样式属性类问题，例如标题层级是否正确、字体字号是否符合要求、参考文献编号顺序是否规范等；第二类是页面版面类问题，例如页码是否居中、页眉是否从指定位置开始、图题表题位置是否正确等。前者更适合从 Word 文档对象本身提取，后者则往往需要结合页面最终呈现结果进行分析。正因如此，本文在系统设计中既使用了 Word 对象分析，也引入了页面分析链路。

### 2.2 python-docx文档对象解析技术

`python-docx` 是 Python 环境下常用的 Word 文档处理库，能够对 `.docx` 文件中的段落、样式、表格和节等对象进行读取和部分编辑。对于本文所研究的毕业论文格式检查任务而言，`python-docx` 提供了较为方便的结构化访问能力，是文档对象分析模块的重要基础。

使用 `python-docx` 时，开发者可以将 Word 文档视为由多个对象层次构成的结构。例如，文档对象包含多个段落和节对象，每个段落又包含多个 run 对象，每个 run 对象可以对应不同的字体和字号属性。通过这些对象，系统可以获取段落文本、标题样式、对齐方式、行距以及部分页眉页脚信息，从而支持摘要、关键词、目录、标题、参考文献等内容的格式检查。

不过，`python-docx` 也存在一定局限。它更适合读取文档逻辑结构和样式属性，而对于一些底层复杂内容，如页码域代码、文本框中的内容、样式继承链以及最终页面布局信息，单靠常规对象接口往往无法完整获取。因此，在本文的实现中，除直接利用 `python-docx` 提供的对象属性外，还结合底层 XML 结构对部分特殊内容进行了补充分析，以提高页码、页脚和页眉等区域信息提取的准确性。

总体来看，`python-docx` 为论文格式检查提供了良好的文档对象分析基础，尤其适合处理标题、段落、字体字号、对齐方式、引文和参考文献等结构与样式属性类规则。

### 2.3 PDF页面文本提取与坐标分析方法

对于页码、页眉、图题表题位置和公式编号位置等版面问题，仅通过 Word 文档对象难以准确判断，因此需要引入页面层面的分析方法。本文选择将 Word 文档转换为 PDF，再从 PDF 中提取页面文本及其坐标信息，以此作为页面分析的基础。

PDF 与 Word 文档相比，更接近文档最终输出时的页面呈现形式。通过 PDF 页面文本提取，可以获得每一页的整体文本内容，以及文本块、文本行或文本 span 在页面中的边界框坐标。这些坐标信息能够反映元素在页面中的实际位置，是判断“是否位于顶部”“是否位于底端居中”“是否位于图下表上”等空间关系的重要依据。

在本文系统中，页面分析阶段主要使用两类信息。第一类是整页文本信息，主要用于页面角色识别，例如判断某一页是否为目录页、摘要页或正文页；第二类是局部文本与坐标信息，主要用于页码候选提取、页眉检测、题注定位以及问题可视化标注。通过同时使用这两类信息，系统能够兼顾整体页面识别与局部版面分析。

因此，PDF 页面文本提取与坐标分析方法是本文实现页面规则检查的重要技术基础。

### 2.4 文档页面版面分析相关方法

文档页面版面分析是指从页面角度研究文档中各类内容的分布方式、空间关系和结构区域。相较于只关注文本内容本身，版面分析更重视文本块、标题、页码、图片、表格和公式等元素在页面中的相对位置。

在论文格式检查任务中，版面分析的意义主要体现在两个方面。其一，某些规则本质上依赖空间关系，例如页码是否位于页面底端居中、图题是否位于图下方、表题是否位于表上方、公式编号是否位于公式右侧行末等。其二，页面整体结构本身也包含较强语义信息，例如目录页通常包含点线连接的目录项，正文首页通常具有“第1章”或“绪论”等起始标记，参考文献页和附录页往往在页面顶部出现明显标题。

本文系统中的页面版面分析方法主要采用基于规则和特征匹配的思路，而不是复杂的深度学习模型。一方面，这是因为当前课题更关注具体规范场景下的可实现性；另一方面，毕业论文的页面结构通常具有较强规律性，利用文本模式、坐标区域和相对位置关系，已经能够支持多数常见版面规则检查。虽然这种方法在复杂文档场景下仍有局限，但对于当前论文格式检查任务具有较好的适用性。

### 2.5 PyQt图形界面开发技术

为了使检查结果更直观地呈现给用户，本文采用 PyQt 构建图形用户界面。PyQt 是 Python 环境下常用的桌面界面开发工具，提供了丰富的窗口控件、布局管理机制和事件响应能力，适合实现具有一定交互复杂度的桌面应用程序。

在本系统中，PyQt 主要用于完成以下几类功能。第一，构建文档选择、规则执行和结果展示等基础交互界面；第二，显示 PDF 页面图像及问题标注结果；第三，实现问题列表与页面跳转之间的联动；第四，支持页面缩放、浏览和结果概览展示。

引入图形界面的意义在于，论文格式检查不仅是“有没有发现问题”，还涉及“问题在哪里、是否便于确认”。如果系统只输出文字结果，用户仍需要手动翻找文档对应位置，使用体验会受到较大影响。通过 PyQt 提供的界面展示能力，本文将检查结果、问题列表和页面可视化标注整合到同一界面中，提高了系统的可用性。

## 第3章 系统需求分析与总体方案设计

### 3.1 系统建设目标

本课题面向毕业设计（论文）格式检查场景，目标是设计并实现一个能够对 Word 文档进行自动检查的软件系统。系统应能够根据学校毕业设计（论文）撰写规范，对论文中的封面、摘要、目录、正文、图表、公式、参考文献等部分进行格式检查，并输出相应的错误提示或警告信息，辅助用户发现论文排版中存在的问题。

传统的 Word 文档格式检查大多依赖文档对象本身的结构信息，例如段落、样式、字体、字号、对齐方式等。这类方法对文本内容和样式属性类问题具有较好的检查效果，但在涉及页面布局和视觉呈现的规则时，往往存在一定局限。例如，页码是否位于页面底端居中、页眉是否从第1章开始启用、章节标题是否另起一页、图题和表题的位置是否正确等问题，仅通过文档对象属性难以准确判断。因此，本系统在文档对象分析的基础上，引入页面分析链路，以增强对版面类规则的检查能力。

结合课题要求与实际实现情况，系统建设目标主要包括以下几个方面：

1. 实现对 Word 文档主要结构内容的自动识别与解析。
2. 实现对文档对象属性类规则的自动检查。
3. 实现对页面布局类规则的自动检查。
4. 实现 docx 检查结果与页面分析结果的统一整合。
5. 实现检查结果的可视化展示与问题定位。

通过上述目标的实现，构建一个兼顾结构信息与页面信息的毕业论文格式检查系统，提高论文格式检查的自动化程度和结果的直观性。

### 3.2 系统功能需求分析

根据毕业论文格式检查任务的特点，系统需要具备文档输入、格式检查、结果汇总和可视化展示等基本功能。结合当前系统实现情况，可以将功能需求划分为以下几个方面。

#### 3.2.1 文档输入与预处理需求

系统首先应支持用户选择待检查的 Word 文档作为输入对象。由于部分页面规则无法直接依赖 Word 文档对象完成判断，因此系统还需要具备将 Word 文档转换为 PDF 页面表示的能力，或者支持用户直接选择已有的同名 PDF 文件参与页面分析。

在文档输入完成后，系统应分别完成两类预处理：一类是对 `.docx` 文档进行结构解析，提取段落、标题、样式等对象信息；另一类是对 PDF 页面进行文本与坐标提取，为后续页面规则检查和问题标注提供基础数据。

#### 3.2.2 文档对象检查需求

文档对象侧主要面向内容结构与格式属性规则。系统应能够对论文中的主要板块进行检查，包括封面、中文摘要、英文摘要、目录、正文、参考文献、附录和致谢等部分。同时，还应能够进一步检查字体、字号、行距、对齐方式、字符间距、标题编号、关键词数量、摘要长度、参考文献编号顺序、引文上标等具体规则。

这部分功能的重点在于利用 `python-docx` 获取到的结构化信息，对论文内容本身的规范性进行判断。

#### 3.2.3 页面分析检查需求

页面分析侧主要面向页面布局与视觉呈现规则。系统应能够识别 PDF 页面中的文本块及其坐标位置，并在此基础上完成以下类型的检查：页码位置与连续性检查、页眉启用边界检查、目录页识别、章节是否另起一页、图题表题相对位置检查、公式编号位置检查以及注释位置辅助检查等。

这部分功能的重点在于结合页面中的文本分布、坐标信息和局部区域特征，对论文的版面格式进行分析。

#### 3.2.4 检查结果展示需求

系统不仅要输出检查结果，还要尽量让结果便于理解。为此，系统需要提供统一的问题列表展示能力，能够对不同来源的检查结果进行汇总，并按照一定顺序编号显示。同时，系统还应支持错误位置的页面可视化标注，使用户能够在页面图像中直接看到问题所在区域，提高结果的可读性和实用性。

#### 3.2.5 用户交互需求

在用户交互方面，系统应支持文档选择、执行检查、问题浏览、页面跳转和页面缩放等基础操作。用户点击问题列表中的某一项后，系统应能够切换到对应页面并在左侧显示相关标注，便于快速定位问题。

### 3.3 系统规则需求分析

论文格式检查规则种类较多，不同规则适合采用的实现方式也不同。为了提高系统的可实现性和检查效果，有必要对规则进行分类分析。

#### 3.3.1 基于文档对象的规则需求

部分规则与文档对象自身的结构和样式属性关系密切，适合通过 Word 文档对象分析实现。例如，摘要字数、关键词数量、标题编号层级、标题标点、字体字号、对齐方式、行间距、参考文献编号顺序、引文上标等，都可以通过提取段落文本和样式属性完成检查。

这类规则的特点是判定依据较明确，对页面视觉布局依赖较小，因此适合作为文档对象侧规则实现。

#### 3.3.2 基于页面版面的规则需求

还有一部分规则本质上依赖页面最终呈现效果，单靠文档对象难以准确判断。例如，页码是否位于页面底端居中、页眉是否从第1章开始启用、目录页的页面特征、章节标题是否位于新页页首、图题和表题的上下位置关系、公式编号是否位于公式右侧行末等。这些规则更适合通过页面文本块、行位置、坐标范围和局部版面关系来判断。

这类规则是本系统页面分析链路的重点内容，也是体现题目中“基于图像处理”特点的重要部分。

#### 3.3.3 混合检查需求

在实际论文格式检查过程中，仅依赖单一方式难以覆盖所有规则。因此系统需要同时支持文档对象分析和页面版面分析，并对两类规则结果进行统一组织。对于页面规则，可直接利用页面号和坐标框进行展示；对于 docx 侧规则，则需要通过文本匹配与页面角色限制等方式进行近似定位，以便在图形界面中实现统一标注。

因此，系统还需要具备混合规则调度、结果合并和统一输出的能力。

### 3.4 系统总体处理流程

根据系统建设目标和功能需求，本文将系统总体处理流程设计为如下几个步骤：

1. 用户选择待检查的 Word 文档；
2. 系统对 Word 文档进行结构解析，构建文档对象上下文；
3. 系统自动寻找同名 PDF，若不存在则执行 Word 转 PDF；
4. 系统对 PDF 页面进行文本与坐标提取；
5. 系统分别执行文档对象侧规则和页面分析侧规则；
6. 系统对两类规则结果进行统一汇总和排序；
7. 系统在图形界面中展示检查概览、问题列表和页面标注结果。

这一流程保证了文档内容检查与页面版面检查能够在同一框架下协同工作，既保留了文档对象分析对样式信息的利用优势，又利用页面分析补足了版面类规则的检查能力。

### 3.5 系统总体架构设计

结合系统处理流程，本文将系统划分为文档对象分析层、页面分析层和混合处理展示层三个部分。

#### 3.5.1 文档对象分析层

文档对象分析层主要负责对 Word 文档进行结构化解析。该层通过文档解析模块提取论文中的各类板块、标题层级、段落文本以及格式属性信息，并将其组织成可供规则调用的上下文数据。随后，系统调用文档对象侧规则完成相关检查，生成相应的问题结果。

该层主要解决内容结构和样式属性类问题，是系统的基础检查层。

#### 3.5.2 页面分析层

页面分析层主要负责对文档的页面表示进行处理。系统先将 Word 文档转换为 PDF，再从 PDF 中提取页面文本及其坐标信息。在此基础上，系统进一步进行页面角色识别，将页面划分为中文摘要、英文摘要、目录、正文、参考文献及附录等不同语义区域。随后，系统执行页面规则，对页码、页眉、章节起始、图题表题、公式编号等版面规则进行检查。

该层主要解决页面布局和视觉呈现类问题，是系统体现页面分析能力的关键部分。

#### 3.5.3 混合处理与展示层

混合处理展示层负责对文档对象侧和页面分析侧的检查结果进行统一组织。该层一方面完成规则调度与结果合并，另一方面负责图形界面中的结果展示与交互。对于页面规则输出的问题，可直接根据页码和坐标进行标注；对于文档对象规则输出的问题，则通过页面角色限制和文本匹配方式实现近似定位，从而在图形界面中进行统一展示。

该层使系统从“能检查”进一步提升到“能直观展示和辅助定位”的层次。

本章围绕系统建设目标、功能需求、规则需求、总体处理流程和总体架构设计，对论文格式检查系统进行了整体分析。通过需求分析可以看出，毕业论文格式检查不仅涉及文档结构和样式属性，还涉及大量页面布局相关规则，因此单一依赖文档对象分析难以满足检查要求。基于此，本文设计了由文档对象分析层、页面分析层和混合处理展示层组成的系统总体架构，为后续系统详细设计与关键功能实现奠定了基础。

## 第4章 系统详细设计

### 4.1 文档对象分析模块设计

文档对象分析模块主要负责对 Word 文档进行结构化解析，并为后续规则检查提供统一的数据基础。由于毕业论文中的大量规则都与段落文本、标题层级、字体字号、对齐方式、行距等对象属性直接相关，因此该模块是系统实现格式检查的基础部分。

在本系统中，文档对象分析模块以 `.docx` 文件为输入，首先读取文档中的段落、节、页眉、页脚等基本对象，再结合已有的文档结构适配逻辑，对论文内容进行分块整理。系统将文档划分为封面、中文摘要、英文摘要、目录、正文、参考文献、附录、致谢等部分，并进一步提取各部分中的标题、正文内容和样式信息。这样做的目的，是将原始文档对象转换为更适合规则调用的结构化上下文。

从实现方式上看，该模块一方面保留了 Word 文档本身的对象信息，另一方面又额外构建了适合检查规则使用的中间表示。例如，对于摘要、关键词、参考文献等内容，系统会提取其对应的段落文本；对于正文中的章节标题，则会提取标题文本及其层级；对于页眉、页脚和页码相关内容，则会额外分析节对象中的头脚区域。通过这种方式，系统既能完成内容结构检查，也能为字体、字号、对齐、行距等样式检查提供依据。

总体而言，文档对象分析模块承担了“从原始 Word 文档到可检查结构化数据”的转换任务，是后续规则引擎正常工作的前提。

#### 4.1.1 文档结构解析设计

论文格式检查的前提，是先识别文档中各部分内容所属的结构位置。因此，本系统在文档对象分析阶段，首先进行文档结构解析。其核心目标是识别论文中的主要板块，并确定各板块内部的标题与正文内容。

在具体设计中，系统依据论文常见的组织方式，对文档中的标题和正文段落进行分析。当识别到“摘要”“Abstract”“目录”“参考文献”“致谢”“附录”等明显结构标记时，系统会将后续相关段落归入对应部分；对于正文部分，则进一步提取章节标题、二级标题等信息，用于后续目录一致性检查、编号层级检查和章节顺序检查。

文档结构解析结果并不直接面向用户显示，而是作为内部上下文供各类规则调用。例如，摘要字数检查规则需要用到中文摘要正文内容，目录一致性检查规则需要用到目录项与正文标题列表，参考文献顺序检查规则需要用到参考文献条目集合。由此可见，结构解析结果在系统中起到了统一数据入口的作用。

#### 4.1.2 文档格式属性提取设计

在完成结构划分后，系统需要进一步提取各类格式属性信息，以支持样式类规则检查。具体包括字体、字号、对齐方式、行距、字符间距以及页眉页脚相关属性等。

对于正文、摘要、目录、参考文献等普通段落，系统主要通过 `python-docx` 提供的段落和 run 对象读取其字体名称、字号大小、段落对齐方式和段落格式信息。对于页眉、页脚及页码等特殊内容，由于其中可能包含域代码、文本框和样式继承关系，系统在读取普通对象属性的同时，还结合底层 XML 结构进行补充分析，以尽量获取更准确的格式信息。

这一设计的意义在于，很多格式问题并不体现在纯文本内容上，而体现在对象属性中。例如，同样是“关键词”三个字，可能存在字体错误、字号错误或对齐方式错误；同样是页码，也可能出现字体、字号、居中方式不符合规范的问题。因此，格式属性提取是样式检查规则成立的基础。

#### 4.1.3 文档对象规则执行设计

在获得结构信息和格式属性信息后，系统进入文档对象规则执行阶段。为了便于扩展和维护，本文将各项检查规则设计为相互独立的规则类，由统一的规则注册与调度机制负责组织执行。

每条规则在执行时都接收同一份文档上下文，并根据自身的检查目标提取所需信息。例如，摘要长度规则只关注摘要文本内容，关键词数量规则关注关键词字段，标题编号规则关注正文标题文本，字体字号规则则关注不同板块中的段落样式。规则执行完成后，统一输出包含规则编号、问题标题、问题描述、问题来源和附加元数据的问题对象。

这种设计方式具有两个优点。第一，规则之间相互独立，后续新增或修改规则时不需要大幅调整已有代码；第二，所有规则结果的数据结构保持一致，便于后续与页面分析结果统一合并和展示。

### 4.2 页面分析模块设计

页面分析模块是本系统的重要组成部分，主要负责补足文档对象分析在页面布局类规则上的不足。由于毕业论文规范中存在大量与最终页面呈现效果直接相关的要求，例如页码位置、页眉出现范围、章节是否另起页、图题表题位置关系等，因此仅依赖 Word 对象属性难以完成准确判断。基于这一考虑，本文引入页面分析模块，将文档转换为页面表示后，再对页面中的文本块与坐标信息进行分析。

与传统意义上完全依赖 OCR 的图像识别方案不同，本系统当前实现更侧重于“页面表示下的版面分析”。也就是说，系统并不是单纯将页面作为图片处理，而是利用 PDF 页面中已有的文本信息及其坐标位置，对页面结构和版面关系进行检查。这种方式在当前课题场景下具有较好的可实现性，也能满足大部分页面类格式检查需求。

#### 4.2.1 Word转PDF处理设计

为了进行页面分析，系统首先需要获得与 Word 文档对应的页面表示。因此，本文在系统中设计了 Word 转 PDF 的处理环节。当用户选择待检查的 `.docx` 文件后，系统会优先查找同名 PDF 文件；若未找到，则调用转换逻辑生成对应 PDF，供后续页面分析使用。

之所以采用 PDF 作为页面分析载体，是因为 PDF 在版面结构上相对稳定，能够较好保留 Word 文档的最终排版结果。与直接从 Word 对象估计页面布局相比，基于 PDF 进行分析更符合用户实际看到的排版效果，也更有利于后续在图形界面中展示问题位置。

#### 4.2.2 PDF页面文本与坐标提取设计

在得到 PDF 文件后，系统需要进一步提取页面中的文本内容及其坐标信息。本文在页面分析模块中，对 PDF 的每一页分别提取文本 span、边界框坐标和整页文本内容，并将这些信息组织成页面对象集合。

其中，整页文本内容主要用于页面角色识别和关键词匹配，而文本 span 及其坐标框则主要用于定位页码、页眉、图题、表题、公式编号等局部元素。与纯文本解析相比，坐标信息能够反映内容在页面中的真实位置，因此在处理页面底端页码、顶部页眉、图表上下关系时尤为重要。

这一阶段形成的数据，是后续页面规则检查和问题可视化标注的核心输入。

#### 4.2.3 页面角色识别设计

为了避免不同页面之间相互干扰，系统在执行页面规则之前，会先对页面语义角色进行识别。所谓页面角色，是指页面在论文整体结构中所处的位置和功能，例如中文摘要页、英文摘要页、目录页、正文页、参考文献页、附录页等。

页面角色识别的设计思路主要基于页面文本特征和局部版面模式。对于目录页，系统会检测“目录”等显式标题，以及点线加页号的目录项分布特征；对于正文起始页，系统会检测“第1章”“第一章”“绪论”等文本模式；对于后置部分，则结合页面顶部标题判断是否进入参考文献、附录或致谢部分。通过这些规则，系统能够对页面进行较为稳定的角色划分。

页面角色识别的意义在于，它可以显著缩小后续规则的检查范围。例如，罗马数字页码应主要出现在摘要部分，阿拉伯数字页码主要出现在正文和后置部分；目录页不应被拿来做正文页眉检查；参考文献内容不应与正文标题定位混在一起。通过先分角色、后检查，系统能够减少误报，提高页面规则的针对性。

#### 4.2.4 页面规则检查设计

在完成页面文本提取和页面角色识别后，系统进入页面规则检查阶段。当前实现中，页面规则主要包括页码存在性检查、页码位置检查、页码样式与连续性检查、页眉内容检查、页眉启用边界检查、章节是否另起页检查、图题表题位置检查、公式编号位置检查等。

这些规则的设计方式与文档对象规则类似，同样采用独立规则类进行封装，但其输入不再只是 Word 对象，而是已经提取好的页面文本与坐标信息。以页码检查为例，系统会在页面底端区域搜索符合页码特征的数字候选，并结合页面角色判断是否应存在页码以及页码应采用哪种编号形式；以图题表题检查为例，系统会根据图表附近文本行与关键词位置关系判断标题是否位于正确的上下方。

总体来看，页面规则检查的核心不在于恢复完整语义内容，而在于利用页面中各类元素的相对位置关系进行格式判断。这也是本系统区别于仅做文本格式检查工具的一个重要特点。

### 4.3 混合规则调度与结果合并设计

本系统采用文档对象分析与页面分析相结合的方式完成论文格式检查，因此在系统内部需要解决两个问题：一是不同来源规则如何统一调度执行，二是不同来源结果如何统一组织输出。为此，本文设计了混合规则调度与结果合并模块。

在规则调度方面，系统首先根据规则注册信息区分文档对象规则、页面规则和混合层辅助规则，然后在统一处理器中依次执行。文档对象规则依赖 `docx` 上下文，页面规则依赖 PDF 页面上下文，混合层则负责后续结果整合和必要的辅助处理。这样的调度方式既保证了不同规则使用合适的数据来源，又避免了规则之间的强耦合。

在结果组织方面，系统将所有规则输出统一转换为问题对象，每个问题对象都包含规则编号、标题、问题说明、严重级别、来源引擎、页码、坐标框和元数据等字段。这样一来，无论文档对象规则还是页面规则，其输出都能够以统一方式进入后续展示流程。

在结果合并过程中，系统还会根据需要对问题进行排序、编号和分组，并过滤掉仅用于状态占位的无效输出，使最终呈现给用户的内容尽量保持简洁和直观。通过这一设计，系统实现了多来源结果的一致化表达。

### 4.4 问题定位与可视化标注设计

为了提高检查结果的可理解性，系统不仅输出文字形式的问题描述，还设计了问题定位与可视化标注功能。其目标是让用户在查看问题时，能够直接看到问题出现在页面中的大致位置，从而减少人工查找的成本。

对于页面规则输出的问题，由于规则本身通常已经能够获得对应页面号和坐标框，因此系统可以直接将其用于页面标注。例如，页码位置问题、图题表题位置问题、公式编号位置问题等，都可以直接在页面图像中绘制标注框。

对于文档对象规则输出的问题，情况相对复杂。因为这类规则往往只知道问题文本属于哪一部分，而不直接包含页面坐标。针对这一问题，系统设计了近似页面定位机制。其基本思路是先根据问题所属板块限制候选页面范围，再在候选页面中进行文本匹配，必要时采用多行窗口匹配和特征文本匹配方式，寻找最可能对应的问题位置。例如，参考文献条目可以在参考文献页中按条目文本匹配，引文问题可以按 marker 进行局部定位。

在可视化展示阶段，系统会将页面渲染结果显示在界面左侧，并叠加问题边框、问题编号等标注元素；用户点击右侧问题列表中的某一项后，界面会自动跳转到对应页面并高亮相关区域。通过这种方式，系统实现了“问题描述”与“页面位置”的联动展示。

### 4.5 图形用户界面设计

为了便于实际使用，本文采用 PyQt 构建了图形用户界面。界面设计目标主要有两个：一是使检查流程尽量直观，二是使结果浏览与问题定位尽量方便。

从整体布局上看，界面主要分为左右两个区域。左侧为页面预览与标注展示区域，用于显示 PDF 页面图像及对应问题框；右侧为控制与结果区域，主要包括文件选择、格式设置、检查概览和问题列表等内容。这样的布局方式符合用户从“执行检查”到“查看结果”的使用习惯。

在交互流程上，用户选择待检查文档后，可以直接触发检查操作。系统完成处理后，会在结果区给出检查概览，并按顺序列出检测到的问题。用户点击任意问题后，左侧页面区域将自动切换到对应页面，并显示相应标注框。若文档中存在较多问题，用户还可以通过页面缩放和翻页功能进行进一步查看。

界面设计虽然不属于格式检查算法本身，但对于系统的可用性影响较大。特别是在毕业论文这类篇幅较长、问题分布较散的文档中，单纯给出文字结果往往难以满足实际使用需求，因此可视化界面是本系统不可缺少的一部分。

### 4.6 本章小结

本章围绕系统详细设计，对文档对象分析模块、页面分析模块、混合规则调度与结果合并模块、问题定位与可视化标注模块以及图形用户界面模块进行了说明。通过上述设计，系统形成了从 Word 文档输入、结构解析、页面分析、规则执行到结果展示的完整处理链路，为后续关键功能实现和系统测试提供了实现基础。

## 第5章 系统关键功能实现

### 5.1 Word文档结构解析实现

系统的第一步是对 Word 文档进行结构解析。为了满足格式检查对结构化数据的需求，本文在实现中首先读取 `.docx` 文档对象，再对其中的段落、节、页眉、页脚等内容进行统一整理。系统在这一阶段并不直接面向界面输出结果，而是构建后续规则可复用的上下文数据。

在具体实现上，系统会优先识别论文中的主要板块，包括中文摘要、英文摘要、目录、正文、参考文献、附录和致谢等。识别过程中主要依据标题文本特征、段落内容和章节组织方式进行判断。对于正文部分，系统进一步提取章节标题及层级信息，用于后续的目录一致性检查、标题编号检查和章节顺序检查。

在完成板块划分后，系统还会整理各部分下的标题文本、正文内容和段落对象，形成适合规则调用的数据结构。这一实现方式的优点在于，后续不同规则不需要重复遍历原始文档，而只需从统一上下文中提取所需内容即可，从而降低了规则之间的耦合度。

### 5.2 页面角色识别实现

为了提高页面规则检查的准确性，系统在页面分析阶段实现了页面角色识别功能。该功能的主要任务是判断每一页属于摘要页、目录页、正文页还是正文后的参考文献、附录、致谢等部分。

在实现中，系统先从 PDF 页面提取整页文本和文本块信息，然后根据页面中的显著文本模式和版面特征进行判断。对于目录页，系统会检测“目录”等显式标题，同时结合目录项常见的点线连接与页号分布模式进行识别；对于正文起始页，系统会重点识别“第1章”“第一章”“绪论”等标题模式；对于后置部分，则通过页面顶部标题是否出现“参考文献”“附录”“致谢”等关键词来确定边界。

页面角色识别结果会被后续多个页面规则复用。例如，页码规则需要区分摘要部分是否应使用罗马数字，页眉规则需要区分哪些页面属于正文部分，问题定位逻辑也需要依赖页面角色缩小搜索范围。因此，该功能虽然实现逻辑相对基础，但在整体系统中具有较强的支撑作用。

### 5.3 典型格式规则实现

由于论文格式规则数量较多，若在论文中逐条展开说明，会导致重点分散。因此，本文选取系统中具有代表性的几类规则进行说明，分别体现文档对象规则、页面规则和混合定位规则的实现方式。

从文档对象规则来看，摘要长度检查、关键词数量检查、标题编号检查、引文上标检查和参考文献编号顺序检查较具代表性。这些规则主要利用文档结构解析结果和段落文本内容进行判断。以摘要长度检查为例，系统通过提取中文摘要内容并计算字符数，判断其是否满足规范要求；以关键词数量检查为例，系统解析关键词字段后统计条目数量，判断其是否在规定范围内。

从页面规则来看，页码位置检查、页眉启用边界检查、章节另起页检查、图题表题位置检查和公式编号位置检查更能体现页面分析能力。以页码位置检查为例，系统会在页面底端区域提取数字候选，并结合页面中心偏移量判断页码是否位于底端居中；以章节另起页检查为例，系统通过页面角色与正文标题分布判断新章是否出现在新页页首。

从混合定位规则来看，文档对象问题的页面定位具有代表性。由于 docx 侧规则本身不直接包含坐标信息，系统需要借助页面角色限制、文本归一化匹配和多行窗口匹配策略，在 PDF 页面中搜索与问题内容最接近的文本区域。该实现虽然属于辅助功能，但对最终界面展示效果影响较大。

### 5.4 文档对象侧规则实现

文档对象侧规则主要围绕内容结构和样式属性展开。在当前实现中，这一部分规则主要包括摘要长度检查、关键词数量检查、标题标点检查、标题编号层级检查、目录一致性检查、参考文献条目格式检查、参考文献编号顺序检查、引文与参考文献一致性检查，以及字体、字号、行距、对齐方式、字符间距等格式检查。

在实现方式上，这些规则均被封装为独立规则类，由统一规则注册机制纳入文档对象规则集合。每条规则在执行时读取统一上下文，并只关注自身相关的数据。例如，目录一致性检查规则需要同时读取目录项和正文标题；引文一致性检查规则则需要同时读取正文中的引文标识和参考文献编号集合。

对于格式类规则，系统不仅检查单个字段是否符合预期样式，还尽量保留问题发生的上下文。例如，当某一段落字体不符合要求时，系统会将相关文本内容作为问题输出的一部分；当标题编号存在层级错误时，系统会将对应标题文本作为出错内容返回。这种实现方式使得检查结果更容易与原文对应，也便于后续可视化定位。

### 5.5 页面版面侧规则实现

页面版面侧规则主要用于解决仅通过 Word 对象难以可靠判断的问题。当前实现的页面规则主要包括页码存在性检查、页码样式与连续性检查、页码位置检查、页眉内容检查、页眉启用边界检查、图题表题位置检查、章节是否另起页检查以及公式编号位置检查等。

在页码相关规则实现中，系统首先从页面底端提取可能的页码候选，再根据数字形式和位置关系进行判断。对于摘要部分，系统重点判断是否使用罗马数字；对于正文及后置部分，系统重点判断是否使用阿拉伯数字以及是否连续；对于目录和第1章前置部分，则重点判断是否错误出现了页面底端页码。为了减少误报，系统在实现过程中还对页码候选区域、候选排序方式以及目录页特殊情况进行了针对性优化。

在页眉相关规则实现中，系统通过提取页面顶部区域文本并与规范页眉内容进行匹配，判断正文页是否缺少页眉，以及页眉是否在第1章前过早出现。该规则直接利用页面顶部文本区域，能够较好反映最终版面效果。

在图题表题和公式编号相关规则实现中，系统主要通过局部文本行的空间关系完成判断。例如，图题应位于图下方且通常居中，表题应位于表上方，公式编号应位于公式右侧行末。系统利用文本块位置和行窗口构造方式，对这些相对位置关系进行分析，从而给出相应提示。

### 5.6 问题定位与可视化实现

为了增强系统的实用性，本文在实现中加入了问题定位与可视化功能。页面规则的问题一般能够直接获得页面号和坐标框，因此定位实现相对直接。系统在提取到这类信息后，会将其记录到问题对象中，供界面层直接绘制标注框。

相比之下，文档对象规则的定位实现更复杂。由于此类规则本质上来源于 Word 文档结构，而非页面坐标，因此系统需要在 PDF 页面中重新搜索对应文本。当前实现中，系统先根据问题所属板块确定允许搜索的页面角色，再对候选页面中的文本行进行归一化匹配；若单行匹配效果不足，则进一步构造多行窗口进行模糊匹配。对于参考文献条目和引文标识等特殊内容，系统还采用了更有针对性的定位方式。

在可视化展示方面，系统会根据定位结果在页面图像中绘制问题框，并为每个问题分配显示编号。右侧问题列表中的条目与左侧页面标注形成联动，用户点击问题条目后，系统会自动跳转到对应页面并高亮相关区域。通过这一实现，检查结果不再只是简单的文字提示，而是能够与论文页面直接对应起来。

### 5.7 本章小结

本章围绕系统关键功能实现，介绍了 Word 文档结构解析、页面角色识别、典型格式规则、文档对象侧规则、页面版面侧规则以及问题定位与可视化等核心功能的实现方式。通过这些功能的协同工作，系统实现了从文档输入、规则检查到结果定位展示的完整处理过程，也为下一章的系统测试与结果分析提供了基础。

## 第6章 系统测试与结果分析

### 6.1 测试环境与测试方法

为了验证系统的可用性和检查效果，本文对已实现的论文格式检查系统进行了功能测试和结果分析。测试内容主要围绕文档对象侧规则、页面版面侧规则、混合结果展示与问题定位等几个方面展开。

测试环境以 Windows 平台为主，系统采用 Python 语言开发，图形界面基于 PyQt 实现，文档对象解析依赖 `python-docx`，页面分析基于 Word 转 PDF 后的页面文本与坐标提取结果完成。在测试过程中，主要以毕业论文格式样本文档及人工构造的异常样例作为测试对象，通过运行系统检查功能，观察问题输出是否符合文档实际情况，并对发现的误报和漏报进行分析。

本文采用的测试方法主要包括以下三种。第一种是规则功能测试，即针对某一条或某一类规则，单独构造存在格式错误的样例，检查系统是否能够识别出对应问题。第二种是整体流程测试，即使用较完整的论文样例运行系统，观察从文档加载、规则执行到结果展示的整体过程是否顺畅。第三种是结果对照测试，即将系统输出与人工检查结果进行比较，分析系统的有效性以及当前仍存在的不足。

### 6.2 测试样本文档设计

为了保证测试结果具有一定代表性，本文在测试时使用了两类文档样本。一类是结构相对完整的论文样本文档，这类文档包含摘要、目录、正文、参考文献等典型板块，适合用于验证系统整体处理流程和多规则协同效果；另一类是人工构造的异常样本，这类样本针对某一类规则故意设置错误格式，用于验证具体规则是否能够稳定触发。

在样本文档设计中，重点关注了以下几类典型问题：摘要长度不足或超出要求、关键词数量不符合要求、标题编号层级不正确、目录项与正文标题不一致、页码位置错误、页眉启用边界错误、章节未另起页、图题表题位置错误、公式编号未位于右侧行末、参考文献编号顺序错误等。通过覆盖这些常见问题，可以较全面地反映系统在毕业论文格式检查场景中的适用性。

此外，为了验证结果定位与页面标注功能，测试文档中还特意保留了部分能够在页面上明显定位的问题，例如页码位置异常、页眉缺失、参考文献条目错误等，以便观察系统在界面中的定位效果是否合理。

### 6.3 文档对象侧规则测试结果

从测试结果来看，文档对象侧规则在内容结构和样式属性类问题上具有较好的表现。对于摘要长度、关键词数量、参考文献编号顺序、引文标识、标题标点和标题层级等规则，系统通常能够稳定地识别出明显不符合要求的情况。

在字体、字号、对齐方式和行距等样式检查方面，系统能够较好地覆盖中文摘要、英文摘要、目录、正文、参考文献等主要板块，并在问题输出中给出对应的出错内容和错误描述。这使得结果不仅能提示“存在错误”，还能够指出问题大致出现在哪一段文字上。

不过，在测试过程中也发现，文档对象侧规则对样式继承、特殊域代码和复杂页脚内容较为敏感。例如，页码格式检查中，如果页码所在对象通过样式链间接继承字号或字体，而不是直接写在当前节点上，就可能出现误判。针对这一问题，系统后续补充了样式链继承处理和页码域 run 信息提取逻辑，从而降低了页码字号误报的概率。由此也可以看出，文档对象规则虽然适合检查样式属性，但在处理 Word 底层复杂结构时仍需结合具体实现细节不断修正。

### 6.4 页面分析侧规则测试结果

页面分析侧规则主要用于检查页码、页眉、章节分页、图题表题和公式编号等版面类问题。从实际测试效果看，这部分规则较好地补足了单纯文档对象分析的不足。特别是在页码位置、页眉内容和章节另起页等规则上，系统能够直接利用页面文本坐标进行判断，因此结果更接近用户最终看到的论文排版效果。

以页码检查为例，系统能够识别摘要部分与正文部分在页码形式上的差异，并检查页码是否位于页面底端居中。在页眉检查中，系统能够根据正文页顶部文本识别规范页眉是否存在，并检测页眉是否在第1章前提前出现。在图题表题和公式编号检查中，系统也能够根据文本块相对位置给出相应提示。

不过，页面分析侧规则同样存在一定局限。测试过程中曾出现目录页中的章节号或目录项页号被误识别为页面页码的情况。这类误报的根本原因在于，目录页本身也包含大量数字和点线连接项，而这些数字在空间位置上与真正页码存在一定相似性。针对这一问题，系统后续对页码候选区域、候选排序方式以及目录页专用过滤条件进行了多次调整，使目录中靠右的条目页号不再轻易被识别为页面页码。经过修正后，相关误报明显减少。

这说明页面规则虽然更适合处理版面问题，但仍然容易受到局部文本布局相似性的干扰，因此在实现上需要结合具体场景不断优化候选筛选条件。

### 6.5 混合检查结果分析

从系统整体运行结果来看，将文档对象分析与页面分析结合起来，能够比单一方式覆盖更多的格式检查需求。文档对象侧规则擅长处理字体、字号、标题编号、引文和参考文献等结构与样式问题，页面分析侧规则擅长处理页码、页眉、章节分页、图题表题和公式编号等版面问题，两者在功能上具有较强的互补性。

在结果展示方面，系统能够将不同来源的问题统一汇总到同一结果列表中，并给出检查概览、问题条目和页面标注信息。与只输出文字问题相比，这种混合展示方式更有利于用户理解问题。例如，用户在看到“页码可能未位于页面底端居中”这类提示后，可以直接在页面预览中查看对应位置，从而快速确认问题是否成立。

同时，混合检查方式也暴露出一个特点：不同来源规则在精度上的表现并不完全一致。文档对象规则通常在结构与样式属性问题上较稳定，但在复杂底层对象场景下可能出现误报；页面规则在版面判断上更直观，但在目录页、复杂文本块布局等场景下可能受到干扰。因此，系统当前更适合定位为“辅助检查工具”，其作用是帮助用户快速发现高概率问题，再结合人工判断完成最终确认。

### 6.6 系统不足与误差分析

尽管当前系统已经能够完成较完整的毕业论文格式检查流程，但从测试结果来看，仍然存在一些不足和误差来源。

第一，部分规则仍然依赖启发式判断，面对格式较特殊或排版方式差异较大的文档时，可能出现误报或漏报。例如，目录页中的数字分布、图题表题附近复杂文本、公式编号与正文混排等情况，都可能影响页面规则的判断结果。

第二，文档对象侧规则在处理 Word 底层复杂结构时仍存在不稳定因素。例如，某些页码、页眉或页脚内容并不是以普通文本形式直接存在，而是通过域代码、文本框或样式继承实现，这会增加格式提取和准确判断的难度。

第三，文档对象问题的页面定位目前仍属于近似定位。虽然系统已经通过页面角色限制、文本归一化匹配和多行窗口搜索等方式提高了定位成功率，但当问题文本较短、重复度较高或页面内容较复杂时，定位结果仍可能不够精确。

第四，当前系统页面分析链路主要依赖 PDF 文本与坐标提取，尚未接入真正意义上的 OCR 图像识别能力。因此，对于扫描版文档、文本提取质量较差的 PDF，系统的页面分析效果还会受到较大限制。

总体来看，这些不足并不影响系统作为论文格式辅助检查工具的基本使用价值，但也说明系统后续仍有较大的优化空间。

### 6.7 本章小结

本章对系统的测试环境、测试方法、测试样本文档、文档对象侧规则测试结果、页面分析侧规则测试结果以及系统整体不足进行了分析。测试结果表明，当前系统已经能够较好地完成毕业论文中的多数常见格式检查任务，尤其在文档结构检查、样式检查、页码页眉检查和问题可视化展示方面取得了较为稳定的效果。同时，测试过程中暴露出的误报与定位误差问题，也为后续系统优化和功能扩展提供了明确方向。

## 第7章 结论与展望

### 7.1 论文工作总结

本文围绕毕业论文格式自动检查需求，设计并实现了一个基于 Python 的 Word 文档格式检查系统。系统以学校毕业设计（论文）撰写规范为依据，围绕论文写作中常见的格式问题，构建了由文档对象分析、页面分析和结果可视化展示组成的完整处理流程。

在系统实现过程中，本文首先基于 Word 文档对象完成了论文结构解析与样式信息提取，实现了对摘要、关键词、目录、标题、参考文献、引文等内容结构与格式属性的检查；随后，为了解决页码、页眉、章节分页、图题表题位置、公式编号位置等难以仅通过文档对象判断的问题，本文引入了页面分析链路，通过 Word 转 PDF 后的页面文本与坐标信息，对页面版面规则进行检查；最后，系统通过图形用户界面将不同来源的检查结果统一展示，并提供页面标注和问题定位功能，提高了结果的直观性和可用性。

从整体上看，本文完成了从规则需求分析、系统架构设计、关键功能实现到测试验证的完整工作过程，较好地达成了课题预期目标。

### 7.2 主要成果总结

通过本课题的研究与实现，取得的主要成果可以概括为以下几个方面。

第一，完成了毕业论文格式检查系统的总体设计与实现，形成了较完整的软件处理流程。系统能够从 Word 文档输入开始，依次完成结构解析、页面分析、规则执行、结果合并和界面展示。

第二，实现了基于文档对象分析的论文结构与样式检查功能。系统能够对摘要、关键词、目录、标题、引文、参考文献等内容进行检查，并支持对字体、字号、对齐方式、行距和字符间距等格式属性进行检测。

第三，实现了基于页面表示的版面规则检查功能。系统能够对页码、页眉、章节另起页、图题表题位置和公式编号位置等页面类规则进行分析，从而弥补了仅依赖 Word 对象分析的不足。

第四，实现了问题定位与可视化展示功能。系统能够将检查结果与页面预览联动展示，使用户在查看问题描述的同时，能够直接观察问题在页面中的大致位置，提高了系统的实用性。

第五，在测试与调试过程中，对部分误报问题进行了针对性优化。例如，页码字号识别、目录页数字误识别为页码、检查结果展示逻辑等问题均得到了一定程度的改进，使系统整体稳定性有所提升。

综合来看，本文实现的系统已经具备一定的毕业论文格式辅助检查能力，能够为用户提供较有价值的自动化支持。

### 7.3 不足与后续展望

虽然本文已经完成了系统的主要设计与实现工作，但受时间、经验和实现条件限制，当前系统仍然存在一定不足，后续仍有继续完善的空间。

第一，当前系统的部分规则仍然采用启发式实现方式，当文档排版形式较复杂或与常规论文样式差异较大时，仍可能出现误报或漏报。后续可结合更多样本文档，对规则条件进行进一步细化和优化，提高系统在复杂场景下的稳定性。

第二，文档对象分析在处理 Word 底层复杂结构时仍存在一定局限。例如，页码、页眉、页脚等区域可能涉及域代码、文本框和样式继承关系，这些情况会影响系统对格式属性的准确提取。后续可以进一步加强对底层 XML 结构的分析能力，以提高规则判断的可靠性。

第三，当前页面分析链路主要依赖 PDF 文本与坐标提取，尚未真正接入 OCR 图像识别能力。因此，当面对扫描版论文、文字提取质量较差的 PDF 或版式更加复杂的页面时，系统的分析能力仍然受限。后续可以考虑引入更完善的 OCR 或文档版面分析模型，以增强系统对复杂页面的适应能力。

第四，当前系统的问题定位仍以近似定位为主，尤其是文档对象侧问题在页面中的位置匹配还存在一定误差。后续可以从文本匹配策略、多区域候选排序以及跨页上下文分析等方面继续改进，提高问题标注的精确度。

第五，当前系统主要针对单一学校毕业论文规范进行实现。后续若能够将规则配置进一步参数化和模板化，则有望扩展到更多学校、更多文档规范场景中，提高系统的通用性和推广价值。

总体而言，本文完成的系统已经为毕业论文格式自动检查提供了一个可运行的实现基础。未来若能在规则精度、页面识别能力、问题定位精度和规范适配范围等方面继续完善，系统的应用价值将进一步提升。
